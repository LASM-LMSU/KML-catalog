# Каталог KML-покрытий ЛАКМ МГУ

Статический каталог для лаборатории аэрокосмических снимков МГУ. Репозиторий хранит исходные `kml`, а сайт на GitHub Pages собирает из них готовый индекс с картой охватов, поиском, фильтрами и технической карточкой записи.

## Что сделано

- `KML` не парсятся на клиенте в реальном времени. На этапе сборки они преобразуются в единый `catalog.json`.
- UI работает как статический SPA на `Vite + React + Leaflet`, поэтому Pages не требует серверной части.
- Каждая запись получает нормализованные поля: сенсор, режим (`standard` / `quicklook`), дата съемки, batch date, bbox, центр, площадь сцены и геометрию полигона.
- Исходные `kml` автоматически копируются в артефакт сборки, поэтому из интерфейса можно скачать оригинальный файл.
- Деплой идет через официальный workflow GitHub Pages.

## Архитектура пайплайна

1. Исходные `kml` лежат в репозитории.
2. Скрипт `scripts/build-catalog.mjs` находит все `*.kml`, кроме build-директорий.
3. `scripts/lib/catalog-builder.mjs`:
   - читает `GroundOverlay` и `gx:LatLonQuad`,
   - извлекает метаданные из имени файла,
   - считает `bbox`, `centroid`, оценочную площадь сцены,
   - собирает итоговый `public/data/catalog.json`,
   - копирует исходные `kml` в `public/catalog-source/`.
4. `Vite` собирает сайт в `dist/`.
5. GitHub Actions публикует `dist/` в GitHub Pages.

## Рекомендуемая структура данных

Текущие исторические папки уже поддерживаются. Для новых поставок лучше придерживаться более предсказуемой схемы:

```text
data/
  batches/
    2026-04-23/
      *.kml
```

Текущий набор KML уже перенесен из корня в `data/batches/2025-12-29/` (внутри сохранена историческая подпапка `Kondor_data_2025-12-29_kml/`).

Скрипт умеет работать и со старой структурой вида `Kondor_data_2025-12-29_kml/`, но для новых загрузок дата в пути вида `YYYY-MM-DD` удобнее: она автоматически попадает в поле партии загрузки.

## Локальная работа

```bash
npm install
npm run dev
```

Полезные команды:

- `npm run catalog:build` — пересобрать индекс из `kml`
- `npm run test` — проверить парсер и базовую нормализацию
- `npm run build` — собрать production-версию
- `npm run check` — тесты + production build

## Как обновлять каталог

1. Добавить новые `kml` в репозиторий.
2. При желании локально проверить `npm run check`.
3. Закоммитить и отправить изменения в `main`.
4. GitHub Actions пересоберет индекс и обновит сайт.

Никакой ручной правки JSON для каталога не требуется.

## GitHub Pages

Workflow лежит в [`.github/workflows/pages.yml`](./.github/workflows/pages.yml). После первого пуша нужно один раз убедиться, что в настройках репозитория в разделе Pages включен режим публикации через GitHub Actions.

Почему такой workflow:

- `actions/configure-pages@v5`, `actions/upload-pages-artifact@v4` и `actions/deploy-pages@v4` соответствуют текущей официальной схеме GitHub Pages.
- `actions/setup-node@v6` с `cache: npm` ускоряет повторные сборки.
- `npm run check` гарантирует, что на Pages уходит только валидная сборка, а не сырые данные.

## Данные каталога

Фронтенд читает один файл:

```text
public/data/catalog.json
```

Формат верхнего уровня:

```json
{
  "schemaVersion": 1,
  "generatedAt": "2026-04-23T09:12:00.000Z",
  "total": 381,
  "bounds": [-37.70, 29.93, 163.21, 80.97],
  "stats": { "...": "..." },
  "records": [{ "...": "..." }]
}
```

У каждой записи есть:

- идентификатор и путь к исходному `kml`
- нормализованные поля из имени файла
- дата и время съемки
- `bbox`, `centroid`, `areaKm2`
- полигон охвата для карты
- поисковая строка для быстрого client-side фильтра

## Дальше можно развить

- Поддержать sidecar-метаданные через `*.json` рядом с `kml` для человекочитаемых названий, регионов, тегов и ссылок на quicklook.
- Добавить автоматическую валидацию duplicate-сцен и предупреждения в CI.
- Если появятся preview JPEG умеренного размера, можно расширить build-step и публиковать миниатюры прямо в карточке записи.
